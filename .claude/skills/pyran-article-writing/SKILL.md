# ぴらん記事ライティング

ぴらん（pyran19）が、ゲーム理論・数理モデルの解説記事を mathlog や note に書くときの文体・構成・図表の扱いを再現し、フォルダ内の素材から1本の記事マークダウンを組み上げるためのスキル。

著者は「対戦ゲームの読み合いを数式でモデル化して遊ぶ」ことを長期テーマにしている書き手で、読者はゲーマー寄り〜数学好き。**数式を出したら必ず日本語の直感で言い換える**、**前提は記事内で完結させる**、**親しみやすく正直なトーン**、この3つが背骨。固い論文調にもAIっぽい説明調にもしないこと。

---

## まず最初にやること（順番を守る）

1. **素材を全部読む。** 解説する内容・調査結果・伝えたいメッセージ・数値結果などは、同じフォルダ（通常 `/mnt/user-data/uploads/`）に置かれている。記事を書き始める前に、そこにあるファイルを必ず先に読む。何があるか分からなければ `view /mnt/user-data/uploads` でフォルダを確認する。素材なしで内容を創作しない。素材が見当たらない場合は、何を記事にしたいのか（テーマ・結果・前提記事）をユーザーに確認する。
2. **参照ファイルを読む。** このスキルの `references/voice.md`（文体と言い回しの辞書）と `references/structure.md`（章構成の詳細テンプレ）に必ず目を通してから書く。下の要約だけで書かない。
3. **設計を固める。** 素材から次を洗い出してメモする ── ①今回の問題設定は何か ②どんなモデル／道具で取り組むか ③得られた結果（主張・数式・表） ④いちばん伝えたいメッセージ ⑤前提となる過去記事や既知の結果（URL含む）。
4. **書く。** 下の骨格に沿ってマークダウンを生成する。
5. **図表のプレースホルダと素材を仕込む。**（後述の3分類に従う）
6. **マークダウンファイルとして出力**し、`present_files` で渡す。

---

## 記事の骨格

連載記事の典型的な並び。すべての章が毎回必要なわけではない（素材の量と種類で取捨する）が、**はじめに → 前提／問題設定 → 計算 → 考察 → まとめ → あとがき**の流れは崩さない。各章の中身と書き方の詳細は `references/structure.md` を読むこと。

1. **概要（任意）** ── mathlog では記事冒頭に数行のアブストラクトを置くことがある。「何を確立した記事か」を簡潔に。
2. **はじめに** ── 署名的な挨拶（例：「こんにちは。…ぴらんです」「ぴら〜ん。…ぴらんです」）→ 今ハマっているゲームなどの軽い近況 → なぜこの問題が面白いか／長期テーマの中での位置づけ。
3. **前提のまとめ／これまでの調査**（過去記事を踏まえる場合）── 後述の「イントロの自己完結ルール」に従う。
4. **やること／やらないこと**（任意）── 箇条書きでスコープを宣言する書き方をよく使う。
5. **要約・結果先出し**（任意）── 「先に結果をまとめます。導出は後ほど」と主要な数式・結論を冒頭に出すことがある。
6. **問題設定** ── 今回解く問題とモデルを、記事内だけで分かるように定義する（自己完結）。
7. **計算** ── LaTeX での導出。小見出しで段階分けし、各ステップの意味を言葉で添える。
8. **適用範囲** ── 公式・結果が成り立つ条件、成り立たない例、落とした近似を正直に書く。
9. **考察** ── 結果が「どういうことを言っているか」を直感に落とす。問いかけ（「どういうことか考えましょう」）を挟みながら小見出しで展開。
10. **具体例**（任意）── 数値を入れた一例を手順通りに追う。マークダウンの表を使う。
11. **まとめ** ── 主要な数式・結論を箇条書き等で再掲。
12. **あとがき／終わりに** ── 「いかがでしたか？」等 → 個人的な感想（「ほくほく」「疲れた〜〜」）→ 読者への感謝 → 今後の予告（「次回は…お楽しみに」）→ 締めの挨拶（「では〜〜」「ではでは〜〜〜」）。

## 文体・トーンの要点

詳細と言い回しの引き出しは `references/voice.md`。最低限ここを外さない：

- **一人称は「わたし」。敬体ベースだが、その中でとても口語的・砕けた語り口。** 読者に直接話しかける。
- **数式 → 直感の順を絶対に守る。** どんな式も、出した直後に「これは要するに〇〇ということ」と日本語で言い換える。「上記の数式が全てですが日本語でも説明すると…」のような橋渡しを入れる。
- **じゃんけん・身近なゲームのたとえ**を多用して概念を導入する。「パソコンくん」「PCくん」のような擬人化、軽いオノマトペ（「ピピピピ…ピコーン！」「ほくほく」「wktk」）も持ち味。
- **正直さと自虐。** 手計算をサボる所は「気合で変形します」、解けなかった所は「解析的に解こうとしたんですが無理でした。数学強い人教えて」と素直に書く。誇張して全部解けたかのように書かない。
- **読者の疑問を先回り。** 「あれ？前にも似た記事書いてなかった？と思う方もいるかも」のように、読者がひっかかりそうな点を自分から拾って答える。
- **概念や評価語は「」で囲む**（「強い」「悪い」など）。本文中の強調は **太字** をたまに使う程度。
- **章の終わりで次章へ繋ぐ。** 「では、計算を進めていきましょう」「では本題です」のような一言で流す。
- 過度な箇条書き・見出し乱発・機械的な丁寧さは避ける。地の文（散文）で語るのが基本で、箇条書きはスコープ宣言・まとめ・要点列挙など効く所に絞る。

## イントロの自己完結ルール（重要）

過去記事や既知の結果を前提にする場合でも、**「前の記事を読まないと分からない」という印象を読者に与えない**こと。具体的には：

- 前提記事は**リンクで引用しつつ、その内容のうち今回使う部分を必ず要約して併記する**。「別の記事を読んでくださいだと不親切なので、今回使う内容をまとめます」という姿勢を地で行く。
- とくに **①問題設定（何を解くのか）と ②どんなモデルで取り組むか は、本文だけで完結させる**。モデルの定義式（例：利得行列、パワー、相性ベクトルの定義）や、引用する過去の結果（公式）は、リンク先に飛ばず本文に再掲して短く意味を説明する。
- 引用は「リンクだけ」で終わらせない。リンク＋2〜4行の要約をセットにする。

## 図表の扱い（3分類）

図・画像は性質で3つに分け、それぞれ別の方法で仕込む。**Claudeがこの場で勝手に数値を計算してグラフを描いてはいけない**（数値図は後で別作業）。判別に迷ったら「その図に実際の計算結果の数字が必要か？」で分ける。

### A. 数値計算が必要な図表 → プレースホルダ＋作業者向けメモ

パラメータを動かしたグラフ、シミュレーション結果、ヒートマップ、計算した数値の散布図など、**実際に数値計算しないと描けない図**。ここでは図を作らず、後で挿入できるように作業者向けの指示ブロックだけ置く。HTMLコメントで挟んで grep しやすくする。

```
<!-- FIGURE:NUMERICAL id="fig-01" -->
> **【数値計算図 / 作業者向け】**
> - 図の意図: （この図で何を見せたいか）
> - 横軸 / 縦軸: （変数と単位・範囲）
> - 描画する曲線・条件: （例: b<c<d のとき、二択部分は点線、実質一択部分は実線、漸近線 y=d, x=b+c-d）
> - 注記したい特徴: （極値・漸近・不連続/微分不可能な点など）
> - キャプション案: *（図の下に付ける一文）*
<!-- /FIGURE:NUMERICAL -->
```

本文側では、この図を指して語る文章はちゃんと書く（図はまだ無くても「下図のように〜」「グラフを見ると〜」と本文は完成させ、図だけ後挿入にする）。

### B. 計算結果を使わない概念的な模式図 → SVGを作成して挿入

ベクトルの位置関係、三すくみの矢印図、原点の取り方、有利方向の説明など、**具体的な数値を必要としない概念図**は、その場で SVG を作って差し込む。

- SVG コードを生成し、出力フォルダの `assets/` に `.svg` ファイルとして保存する（例：`assets/concept-01-sukumi.svg`）。
- マークダウン本文には画像参照とキャプションを置く：`![三すくみの関係](assets/concept-01-sukumi.svg)` 直後に *キャプション* を1行。
- 見つけやすいよう直前に `<!-- FIGURE:SVG id="..." 概念図の一言説明 -->` を添える。
- SVG はシンプルで読みやすく。座標軸・原点・ベクトル矢印・点ラベルを最小限の色（黒＋差し色1色程度）で。装飾過多にしない。図中の文字は日本語可。
- 注記: mathlog / note では画像をアップロードする運用になることが多い。SVG ファイルはそのままアップロード、もしくは PNG に変換して使えばよい旨を、納品時に一言添える。

### C. サムネイル／イントロの薄いイメージ図 → 画像生成AI用プロンプトを用意

記事の雰囲気・メッセージをふわっと伝えるサムネイルやアイキャッチ（具体的な数式や図ではなく、抽象的・情緒的な「薄い」イメージ）は、diffusion 系の画像生成AIで別途作る。ここでは**生成用プロンプトだけ用意し、挿入箇所は空けておく**。

```
<!-- IMAGE:THUMBNAIL -->
> **【サムネイル / イメージ図 — 画像生成AI用プロンプト】**
> Prompt: "（英語で。抽象的・概念的・ミニマルで、記事テーマの雰囲気を喚起する情景。例: minimalist abstract illustration of rock-paper-scissors as three balanced vectors on a soft gradient field, conceptual, clean, muted palette, no text）"
> 補足: アスペクト比・色味・避けたい要素など
> （↓ ここに生成画像を挿入）
<!-- /IMAGE:THUMBNAIL -->
```

プロンプトは英語で書く（多くの画像モデルが英語前提）。記事の主題を直訳的に描かせるより、**抽象度の高い・静かで美しい・文字なし**の方向に寄せると「薄いイメージ図」になる。

## 数式の作法

- インラインは `$ ... $`、独立式は `$$ ... $$`。複数行の導出は `\begin{align*} ... \end{align*}`。
- mathlog / note のどちらでもレンダリングされる標準的な LaTeX を使う。行列は `\begin{pmatrix}` または `\pmatrix{}`。
- 連載で確立した記法を踏襲する（毎回これに縛られはしないが、同じ枠組みを扱うなら必ず合わせる）：利得行列 $A_{ij}=p_i-p_j+x_iy_j-y_ix_j$、パワー $p$、相性ベクトル $v=(x,y)$、2次元の外積 $v_i\times v_j$、混合戦略・ナッシュ均衡・三すくみ など。同じ量には同じ文字を使い、記事内で初出時に必ず定義する。
- 導出は「分母／分子」「第一項／第二項」のように分割し、各ブロックの後に何が起きたか（例：「計算の結果 $p$ 部分が消えました」）を書く。式だけ並べない。

## 出力

- 1本の記事を1つの `.md` ファイルとして `/mnt/user-data/outputs/` に保存する。ファイル名は内容が分かる日本語またはローマ字で（例：`構築同士の相性関係.md`）。
- 概念図 SVG を作った場合は `/mnt/user-data/outputs/assets/` に置く。
- `present_files` で記事 `.md`（先頭）と、生成した SVG があればそれらを渡す。
- 納品メッセージは簡潔に。図表プレースホルダ（数値図／サムネイル）が何箇所あり、何を後挿入すればよいかを一言添えると親切。長い後書きは不要。

## 参照ファイル

- `references/voice.md` ── 文体・口調・言い回しの詳細な引き出し（挨拶/締めの定型、文末の癖、たとえ、自虐表現、やってはいけない書き方）。書き始める前と、トーンに迷ったときに読む。
- `references/structure.md` ── 章ごとの目的・書き方・実例での対応、イントロ自己完結のさらに詳しい解説。構成を組むときに読む。
